## Codebase Patterns
- Keep the root README focused on install/use/configure quickstart flow; move longer operational recipes with infrastructure snippets into `docs/` and link from the relevant README sections.
- Keep `datafast_session_id` capture limited to Creem metadata until DataFast documents payment API support for it; webhook payment payloads should forward only `datafast_visitor_id`.
- Treat Creem initial subscription `checkout.completed` deliveries as control-flow only; ignore them and let `subscription.paid` own payment attribution so the first recurring purchase is not forwarded twice.
- Verify Creem webhooks against the raw body and keep adapters transport-only; Express integrations must use `express.raw({ type: "application/json" })`.
- Keep tracking precedence explicit: `params.tracking` overrides metadata tracking keys, metadata tracking keys override `request.url` query params, query params override cookies, and only `datafast_visitor_id` / `datafast_session_id` are ever injected.
- Map `refund.created` to a fresh DataFast payment with `refunded: true` and use the Creem refund id as `transaction_id` so partial/full refunds keep unique event identities.
- `subscription.paid` should hydrate `last_transaction_id` when available and only fall back to product pricing on hydration failure.
- Creem webhook `object.customer` fields are polymorphic; preserve `customer_id` whether the payload sends a hydrated object or a raw string id.
- DataFast forwarding should use explicit per-request timeouts and bounded retries in the core client; retry only network errors, timeout aborts, and HTTP `408` / `429` / `5xx`.
- Workspace examples should import the built root package through published `exports`; rebuild the root package before running the example after library changes.
- Keep `creem-datafast` root exports limited to the framework-agnostic core; adapter/browser APIs and their types should stay on subpath entrypoints.
- Runtime error classes that consumer code may branch on should be exported from the root package, not only from internal files.
- Next.js-specific webhook conveniences belong in `creem-datafast/next`; if they accept `Request`, document clearly that they consume the body stream once.
- Creem transaction hydration fixtures should use millisecond `createdAt` values to match the SDK and catch timestamp unit regressions.
- Distribution confidence should come from installing the packed `.tgz` into an isolated consumer fixture and checking root plus subpath imports, not from importing source files inside the repo.
- Keep root-package CI separate from framework-example CI when their runtime floors differ; the library's Node support claim should be validated independently from `example-next`'s Next.js minimum.
- Workspace examples should import the built root package via `workspace:*` and `exports`, not source aliases to `../src/*`; packaged smoke tests already cover the tarball case.
- Lightweight adapter demos can stay as small TypeScript workspaces run with `tsx` plus a local `.env.local` loader; avoid pulling in bundlers or `dotenv` unless the example truly needs them.
- Test tooling must also honor the package Node floor; if CI validates Node 18, keep `vitest` on a major that still supports Node 18.
- Keep GitHub Actions core actions updated when GitHub deprecates their runner Node version, or CI will stay green but accumulate avoidable platform warnings.
