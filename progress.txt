## Codebase Patterns
- Verify Creem webhooks against the raw body and keep adapters transport-only; Express integrations must use `express.raw({ type: "application/json" })`.
- Keep tracking precedence explicit: `params.tracking` overrides metadata tracking keys, metadata tracking keys override cookies, and only `datafast_visitor_id` / `datafast_session_id` are ever injected.
- Map `refund.created` to a fresh DataFast payment with `refunded: true` and use the Creem refund id as `transaction_id` so partial/full refunds keep unique event identities.
- `subscription.paid` should hydrate `last_transaction_id` when available and only fall back to product pricing on hydration failure.
- Creem webhook `object.customer` fields are polymorphic; preserve `customer_id` whether the payload sends a hydrated object or a raw string id.
- The Next example maps `creem-datafast` subpaths to `../src/*` via TS paths so it can typecheck and build without depending on generated `dist/`.
- Keep `creem-datafast` root exports limited to the framework-agnostic core; adapter/browser APIs and their types should stay on subpath entrypoints.
- Next.js-specific webhook conveniences belong in `creem-datafast/next`; if they accept `Request`, document clearly that they consume the body stream once.
- Creem transaction hydration fixtures should use millisecond `createdAt` values to match the SDK and catch timestamp unit regressions.
- Distribution confidence should come from installing the packed `.tgz` into an isolated consumer fixture and checking root plus subpath imports, not from importing source files inside the repo.
- Keep root-package CI separate from framework-example CI when their runtime floors differ; the library's Node support claim should be validated independently from `example-next`'s Next.js minimum.
- Workspace examples should import the built root package via `workspace:*` and `exports`, not source aliases to `../src/*`; packaged smoke tests already cover the tarball case.
